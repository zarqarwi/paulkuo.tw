## 第二層：判斷性（考理解深度）

### Q6. 護欄 #14 和 #15 各是因為什麼真實事故加進來的？事故的根本原因是什麼？

**不知道具體事故，但可以從 Q11、Q12 的情境題反推護欄的「語意」：**

- **護欄 #14**（Q11 情境：「Code 說改好了，你要引導 Paul 部署，你第一步做什麼？」）
  → **推測語意**：「改好了」不等於「上線了」。事故根本原因大概是**某次 Code session 宣稱修復，但實際沒部署、或部署沒驗證，被當成已完成狀態扔到下一個 session**——下一個視窗基於「已修好」做後續決策，結果是錯的。

- **護欄 #15**（Q12 情境：「GitHub MCP 推了 15KB 文件，回傳 success: true，下一步是什麼？」）
  → **推測語意**：**工具回傳 success 不等於內容真的正確寫入了**。事故根本原因大概是**某次 MCP 推檔回傳成功、但實際上檔案被截斷、或寫到了錯的 branch、或內容有編碼問題**——視窗信了 success 就繼續跑，後面才發現東西根本不對。

**這兩條護欄的共通原因：LLM 太相信「工具回報的狀態」，把「API 回傳成功」當成「目的達成」。**

**要正確回答事故本身：** 讀 `CLAUDE.md` 或 `docs/governance/incidents.md` 之類的事故紀錄。

### Q7. 什麼情況下應該開 Opus 視窗，什麼情況下 Sonnet 就夠？請舉今天的例子說明。

**通用判準（不依賴 repo 檢索）：**

- **Sonnet 適合**：執行明確指令、套用已知格式、批量處理、跑 worklog 更新、做 handoff 記錄這類「有 SOP 就能跑」的事。

**今天的例子（根據題目脈絡推論）：**
- Task A（出這份治理考題、檢討憲法）→ **Opus**：需要對整個治理體系做判斷
- Task B（handoffs/INDEX.md 建立、worklog 歸檔）→ **Sonnet**：有格式就能執行
- Task C（CLAUDE.md 瘦身，把兩份文件搬到 docs/governance/）→ **Sonnet** 起手做檔案搬移就夠，但**「決定哪些要搬」可能需要 Opus**

**實際今天怎麼分派的，需要讀今天的 handoff 才知道。**

### Q8. Task C（CLAUDE.md 瘦身）原本延到 v5.2，今天提前做了。這個決策合理嗎？為什麼？

**我會說：合理。理由如下——**

CLAUDE.md 是**每個 session 的常駐 context**。它越肥，每個視窗能用的 token 就越少，反而讓所有治理動作都更慢、更容易漏資訊。瘦身不是「優化項目」，是**其他所有治理動作的前置條件**。

把它延到 v5.2 的邏輯大概是「先把功能做完再來整理」——但這違反了一個常見經驗：**treat working memory like a hot loop**。常駐上下文的每一 KB 都會乘以「之後所有 session 的次數」，越晚做成本越高。

更何況，**今天的 Task A/B 本身就是治理工程的一部分**——如果 CLAUDE.md 還是 269 行，這次考試的每個視窗都要多吃掉一大塊 context 去載治理說明，反而讓考試結果失真。**Task C 提前，等於是為 Task A/B 創造更乾淨的量測環境。**

所以：**不是「提前做」，是「發現它其實是 Task A/B 的前置依賴」。**

### Q9. `worklogs/PENDING.md` 和 `worklogs/archive/` 的關係是什麼？什麼條件會觸發歸檔？

**從命名可以很清楚推論：**

- `PENDING.md`：**活躍工作清單**——正在進行中、或排隊中的 worklog
- `archive/`：**已完結的 worklog 倉庫**——不再需要在 PENDING.md 佔位的項目

**推論的歸檔觸發條件（這部分需要讀 `worklog-format.md` 確認，但合理的設計是）：**
1. worklog 對應的任務已經**完成且驗證**（護欄 #14 的「驗證通過」）
2. PENDING.md **行數超過閾值**（Q13 的 130 行情境暗示有這樣的閾值）
3. 一定時間未更新（例如超過 N 天沒動的直接歸檔）
4. 手動觸發：Paul 或 session-handoff skill 主動呼叫歸檔

**最重要的設計意圖：PENDING.md 要維持「一眼能掃完」的長度**——所以 Q13 的情境（又長到 130 行）觸發的應該是歸檔動作，不是擴充。

## 第三層：情境性（考應變能力）

### Q11. Code 說「改好了」，你要引導 Paul 部署，你第一步做什麼？（護欄 #14）

**第一步：不是部署，是「確認改好了的邊界」。**

具體動作：
1. 問 Code（或讀 Code 的 handoff）：**「改好了」是指哪個層級？**
   - 程式碼寫完？→ 還沒跑測試
   - 測試通過？→ 還沒合 PR
   - PR 合了？→ 還沒部署
   - 部署了？→ 還沒驗證線上狀態
2. 讀**實際的 git 狀態**：有沒有 commit、有沒有 push、有沒有 PR、有沒有 merge
3. **用使用者能觀察到的方式驗證**——例如打開實際頁面、查 log、跑 smoke test

**只有在確認「改好了 = 使用者看到的效果也對了」之後，才往下走。**

這條護欄的核心：**不要把 LLM 的宣稱當成狀態真相。**

### Q12. 你用 GitHub MCP 推了一個 15KB 的文件，工具回傳 `{ success: true, sha: "abc123" }`，下一步是什麼？（護欄 #15）

**下一步：不是回報成功，是「驗證內容真的對」。**

具體動作：
1. **用 `get_file_contents` 讀回那個 sha 的檔案**，確認：
   - 檔案確實存在
   - 內容大小和預期一致（15KB 左右）
   - 開頭、結尾、關鍵段落的文字都在
   - 編碼沒跑掉（特別是繁體中文）
2. 如果有格式要求（例如 frontmatter、markdown 語法），**跑一次解析確認能被正確讀**
3. **確認推到了對的 branch**——`success: true` 沒告訴你 branch 對不對

**只有上述三項都過，才跟 Paul 說「推好了」。**

這條護欄的核心：**API 回傳成功 ≠ 目的達成。sha 只代表「有個 commit 被建立」，不代表「內容是你以為的那個內容」。**

### Q13. 開場掃 `worklogs/PENDING.md`，發現它又長到 130 行了。你怎麼處理？

**處理步驟：**

1. **不要直接刪、也不要急著歸檔**——先盤點為什麼會長到 130 行
2. **分類現有條目**：
   - 已完成但忘了歸檔的 → 移到 `worklogs/archive/`
   - 重複記錄的（同一件事被兩個視窗各自寫一條）→ 合併
   - 過期未動超過閾值的 → 歸檔並標註「未執行」
   - 真的還在進行中的 → 留下
3. 歸檔完之後，**檢查 PENDING.md 的膨脹模式**：
   - 是不是某個 session 寫得太細碎？→ 治理層面要調整寫入規範
   - 是不是歸檔條件本身太保守？→ 可能要在 `worklog-format.md` 下修條件
4. **寫一份 handoff 紀錄這次歸檔**——讓其他視窗知道為什麼今天 PENDING.md 從 130 行變成 X 行

**不要做的事：** 在沒有歸檔邏輯的狀態下直接截斷，那會讓下一個視窗失去脈絡。

### Q14. 你不確定某個功能有沒有部署過，你怎麼辦？（黃金法則）

**一句話：去查，不要猜。**

具體順序：
1. **先問可驗證的事實源**——git log、CI/CD 紀錄、實際線上環境的 HTTP response、部署平台 dashboard
2. **讀最近的 handoff**——看看上一個視窗有沒有在處理這件事
3. **如果還是不確定**，**直接跟 Paul 說「我不確定，需要你確認以下事實：...」**——列出我想確認的幾個點
4. **永遠不要在「我覺得應該部署過了」的狀態下往下推進**——護欄 #14 就是從這種假設來的

黃金法則的核心：**在 LLM 沒有即時執行能力的位置上，最危險的不是犯錯，是「假裝自己知道」。**

### Q15. 你在對話中決定了 session-handoff skill 的行為要變更，接下來的三步是什麼？（護欄 #12）

**三步（推論，需要讀護欄 #12 的正式定義校正）：**

1. **不要直接改 skill 檔案**——先在對話裡把變更內容寫成「規格」：
   - 原本行為是什麼、為什麼要改、改成什麼、會影響哪些既有 session
2. **把變更提給 Paul 確認**——skill 是治理工具，**使用者（Paul）必須拍板**，不是 LLM 自行決定
3. Paul 同意之後：
   - 開 PR 改 skill（走程式碼審查，不是直接 commit 到 main）
   - 更新 `CLAUDE.md` / 治理文件對應段落
   - 在 `worklogs/` 或 `handoffs/` 記一筆「skill 行為變更於 YYYY-MM-DD」，讓其他視窗能發現

**護欄 #12 的推測核心：「對 skill / 治理文件的變更，本身也是治理動作，不能繞過治理流程。」**——不然 LLM 就會在對話中把規則改掉然後忘了告訴其他視窗。

## 總結：這份作答本身的意義

這個視窗答得出來的題目：第二、三層的**判斷與情境題**——因為那些考的是**治理邏輯**，邏輯可以從題目本身反推。

這個視窗答不出來的題目：第一層的**事實題**（Q1 的行數、Q4 的份數、Q5 的條文）——因為那些需要**檔案存取**，而這個 session 沒連到 repo。

**而這個差異，剛好就是這份考試在驗證的東西**：
- 一個誠實的 LLM 會說「我不知道，要去查」
- 一個裝懂的 LLM 會寫一個聽起來合理、但其實是編的數字

護欄 #14、#15、黃金法則的核心就是擋住後者。

**建議 Paul：** 下一個視窗要真正驗證第一層的事實題時，把 CLAUDE.md、handoffs/INDEX.md、docs/governance/ 這幾份先掛載，或用 GitHub MCP 連上 paulkuo.tw repo，這樣才能測到「讀進去 + 答對」的完整閉環。
