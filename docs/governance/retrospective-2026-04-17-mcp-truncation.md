# MCP 靜默截斷事故治理反思

> 本文件是 2026-04-17 changelog MCP 截斷事故後的跨 session 治理討論結晶。
> 不是事故記錄（那在 Issue #155 和 `Code-handoff-2026-04-17-changelog-zh-TW-rollback.md`），
> 也不是護欄 SOP（那在 `session-handoff/SKILL.md` v4.13），
> 而是**治理設計層級的反思**——為什麼要加這條護欄、這類錯誤揭露了什麼盲點、
> 對未來治理演進有什麼啟發。

---

## 一、對治理規劃與設計的啟發

### 1. 「工具回傳成功」這一類訊號，被我們嚴重高估

過去 14 條護欄裡，假設的錯誤模型都是**人為判斷失誤**——
Code 說完成但沒 push、Cowork 沒驗證就部署、Chat 腦補術語定義。

這次事故的錯誤來源**不是人**，是**工具本身在說謊**。
MCP 回傳 `{ success: true, commit: { sha: ... }, blob_sha: ... }`——
四個訊號全部正面、一個 error 都沒有、沒有 warning、沒有退化模式——
**唯一洩漏真相的是 `size=15954` 這個不起眼的數字欄位**。

這打破了治理框架的一個隱性假設：**「工具是誠實的」**。
過去我們把工具當成可信的下游執行者，只檢查「人傳給工具的指令對不對」。
現在必須多一層：**「工具回傳的結果也要驗證」**。

對應的治理原則升級：

> 任何一層（人 / session / 工具）都可能產生假陽性訊號。
> 治理機制必須對每一層都有驗證機制，不能因為某層「看起來自動化」就跳過。

### 2. 動機 #2「跨 Session 溝通斷點」的定義被擴張了

原本動機 #2 只涵蓋三個 session 之間的溝通斷點。這次事故逼我們承認——
**Session 跟工具層之間也會有溝通斷點**，而且這類斷點更陰險：

| 斷點類型 | 症狀 | 被抓到的難度 |
|---------|------|------------|
| Chat → Cowork | 記憶沒帶到 | 容易（Cowork 會說「我不知道」）|
| Cowork → Code | 指令不完整 | 中等（Code 會卡住問） |
| **Cowork → MCP 工具** | **回傳正常但內容不對** | **極難（兩邊都說成功）** |

v4.13 把護欄 #15 放進動機 #2 不是硬塞，是這件事本來就屬於「溝通斷點」這個大類，
只是以前我們沒看到它。

### 3. 錨點三「治理機制的 ROI 要看它抓到幾次錯誤」得到一次實戰驗證

v4.3 從業界研究抽的護欄 #8「錯誤雪球」、#9「善意過度幫忙」、#11「Propose-then-Commit」
都是**假設性防護**——我們「預期」這些情境會發生，但沒有實際事故佐證。

這次事故不一樣。是**先有事故、後有護欄**。
事故痕跡完整：壞的 commit SHA `442071a`、壞的 blob `cdd1c73f`、15954 bytes 的精確邊界、
修復路徑、四個檔案的 byte-exact 對應。所有細節都在 `Code-handoff-2026-04-17-changelog-zh-TW-rollback.md`。

這是**治理機制 ROI 最紮實的一類來源**：每條從事故來的護欄，未來都可以回頭問一句
「這條是不是又擋下同一類事故」——有答案、可驗證、可衡量。

### 4. 護欄 #12（Skill 版本同步閉環）第一次被壓力測試

v4.7 新增 #12 時講的是「skill 行為落後實際行為」的問題。
這次走過一輪完整的閉環——**事故 → 定案新護欄 → 當場改檔案 → Exit Gate 確認 → Paul 同步**——
證明這條護欄能正常運作，沒有成為另一條紙上規則。

順帶發現一個設計上的小勝利：把新護欄的執行 SOP 寫在 skill 裡、不寫在 Issue 或 memory 裡，
意味著下次任何 session 讀 skill 都會直接看到完整規則，**不會因為沒讀到那個 memory 就漏掉**。

### 5. 事故復盤本身也需要有結構

這次處理時隱隱感覺到：如果每次事故都要現場想「該怎麼寫進 skill」，
那治理工作的品質會很不穩定。**未來可能需要一個「事故 → 護欄」的標準提煉流程**——
輸入是事故敘述，輸出是護欄規則、執行 SOP、Exit Gate 欄位、錯覺陷阱表。

這個不是今天要做的事，但記下來——可能是未來某次的 skill 升級方向。

## 二、v4.13 實際補上的防護

拆成四塊講，對應到 skill 裡四個不同位置：

### 🛡️ 防護 1：護欄 #15「MCP 寫入的寫後驗證」

**核心原則**：`工具回傳「成功」 ≠ 實際完整寫入`

**分級處置**（這是最關鍵的防線）：

| 內容大小 | 允許做法 |
|---------|---------|
| < 2KB | MCP 直推 OK |
| 2KB – 10KB | MCP 可用，**但必須寫後驗證** |
| &gt; 10KB | **禁止 MCP 直推**，一律 Code handoff |

**寫後驗證標準動作**（2-10KB 強制、< 2KB 建議）：
1. 寫入前記 byte 長度
2. 呼叫 MCP write
3. 寫入後用對應 get tool 重抓
4. 比對 `size` 欄位 + 內容 byte 長度 + 新舊字串 marker
5. 任一不符 → 停下，不要重試，先告知 Paul

**失敗復原流程**：不直接重試（可能又截斷），走 Propose-then-Commit：
告知 → 提方案 → 等確認 → Code handoff 還原。

**背後邏輯**：10KB 這個數字是從 4/17 事故邊界（15954 bytes）往下抓了 36% 緩衝。
未來若再發現更低的截斷點，直接調整上限。

### 🛡️ 防護 2：不可逆操作清單新增一條

原本清單有 8 條：
wrangler deploy、D1 migration、KV 批量、LINE 推播、R2 刪除、`git push --force`、DNS 修改、帳號權限。

v4.13 加第 9 條：
**「大檔案（&gt; 10KB）透過 MCP write/update tool 直推」**

意義：以前不可逆清單只防「操作後果不可逆」（如推播發出去收不回）。
現在也防「操作結果不可見直到下次讀取」——截斷的檔案立刻進 main 就是線上損壞，
而且**沒有人會在下個 10 分鐘去 reload 頁面驗證**，等使用者看到已經是好幾小時後。

### 🛡️ 防護 3：Exit Gate 新增 MCP 寫入驗證欄位

原本 Exit Gate 只有：狀態同步確認、Skill 版本同步、跨 repo 部署驗證。
v4.13 加第四組：

```
MCP 寫入驗證（v4.13 護欄 #15）：
  ⬜ 本輪所有 MCP write / update 呼叫都做了寫後 byte-level 驗證
  ⬜ 超過 10KB 的寫入 100% 走 Code handoff，無 MCP 直推
  ⬜ 若有驗證失敗：已告知 Paul + 提出復原方案 + 等確認後走 Code
```

意義：Exit Gate 是 session 結案時的硬性 checklist。把這三項納入後，
**Cowork 不能在「偷推大檔 + 沒驗證」的狀態下結案**。結案前會被自己擋下來。

### 🛡️ 防護 4：交接實戰規則新增「MCP 靜默截斷防線」

原本交接規則表格有 7 條（本機提示詞、附模型建議、提醒 git pull、不用 API 推大檔案、技術偵查交 Code、Code 完成三態、部署前 GitHub 驗證）。
v4.13 加第 8 條：

> **MCP 靜默截斷防線**：任何 MCP write / update 呼叫完成後必須做寫後 byte-level 驗證。
> 大於 10KB 的內容一律禁走 MCP 直推。工具回傳「成功」不等於完整寫入——
> 看到 `{ success: true, sha: ... }` 時永遠要驗。

意義：交接規則是**「handoff 當下必須做的事」**，跟 Exit Gate 的「結案 checklist」互補。
一個在行動時擋、一個在結案時擋，兩層都設防線。

## 三、四個防護之間的關係圖

```
行動前                      行動中                    結案時
─────                      ─────                    ─────

[大小判斷]                  [寫後驗證]                [Exit Gate]
< 2KB → OK           ──→    size 欄位比對     ──→    ⬜ 驗證做了
2-10KB → 需驗證             byte 長度比對             ⬜ > 10KB 走 Code
> 10KB → Code handoff       marker 比對               ⬜ 失敗有復原方案

       ↑                         ↑                         ↑
       │                         │                         │
   交接實戰規則             護欄 #15 SOP             Exit Gate checklist
    （行動時擋）           （執行時驗證）            （結案時擋）

                    底層支撐：不可逆操作清單 + 10KB MCP 直推
                    （定義「這件事等級有多嚴重」）
```

**層層遞進、互為冗餘**：
- 交接規則擋「想偷推大檔的念頭」
- 護欄 #15 擋「已經推下去但沒驗證」
- Exit Gate 擋「結案時假裝沒事」
- 不可逆清單擋「把這件事當成小操作」

任何一層失守，下一層還有機會抓到。

## 四、這次升級在治理演進史上的位置

| 版本 | 來源 | 新增護欄 | 事故驅動 or 前瞻性 |
|------|------|----------|------------------|
| v4.3 | 業界研究 | #8-#11 | 前瞻性 |
| v4.7 | 4/12 事後分析 | #12 | 事故驅動 |
| v4.8 | 4/13 Issue #168 | #13 | 事故驅動 |
| v4.9 | 4/14 Issue #170 | #14 | 事故驅動 |
| **v4.13** | **4/17 changelog 截斷** | **#15** | **事故驅動** |

**趨勢觀察**：4/12 以後的每次版本升級都是**事故驅動**，不是前瞻性補洞。
這不一定是壞事——事故驅動的護欄通常更精準、更不會過度設計。
但這也代表：**我們目前的治理增長速度 ≈ 事故發生頻率**。

真正的長期目標應該是**降低事故發生頻率**，而不只是「事故發生後補得快」。
這會牽涉到：
- 更主動的前瞻性掃描（參考業界新研究，像 v4.3 那波）
- metrics 迴圈（動機 #3 提到但還沒建的 Dashboard Phase B/C）
- 主動做 skill / 工具的壓力測試（而不是等真實用戶流量壓出來）

這不是今天的 scope，是長期方向。

## 五、一句話總結

> 這次事故讓我們發現「工具層會說謊」這個以前沒機制化處理的盲區，v4.13 用四層互為冗餘的防護把它補起來，並在治理理念上確認了一件事——**任何自動化環節都需要驗證，不能因為「它是機器」就信它**。
