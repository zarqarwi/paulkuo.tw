# L3 操作 SOP 定位釐清 — 維持現狀 + 寫作規則

## Context

### 立法動機

2026-04-24 Cowork 治理架構回顧 artifact（`paulkuo-governance-architecture-2026-04-24`）把治理分層分為 L1-L5 共五層：

- **L1**：憲法（`adr-collaboration-constitution-v0.2-2026-04-19.md`）
- **L2**：實施細則（`docs/governance/working-environment.md`，§1-§9）
- **L3**：操作 SOP（`docs/governance/working-environment.md` §5「操作 SOP」+ 速記卡）
- **L4**：skill 規範（`.claude/skills/**/SKILL.md`）
- **L5**：基礎設施（hooks、scripts、CI）

**問題**：L2 和 L3 **物理上是同一個檔案**——`working-environment.md` §1-§4 + §6-§9 是 L2（原則性實施細則），§5 是 L3（操作 SOP，2026-04-24 新增整章）。架構圖把 L3 拉出來視為獨立層，但它其實是 L2 檔案的子章節。

讀者看到 L2 / L3 並列，可能誤以為：
- L3 有獨立檔案（實際上沒有）
- L2 vs L3 有嚴格的內容邊界（實際上 §5 是 L2 文件的一部分）
- 寫新規則時要判斷「該寫 L2 還 L3」（實際上決策空間是「該寫 §1-§4 還 §5」）

**架構圖出現邏輯重疊**，而讀者若不看檔案結構，直接看圖會混淆。

### 歷史脈絡：§5 新增的來源

`working-environment.md` §5「操作 SOP」整章是 2026-04-24 rev2.3 新增：

- **§5.1**：多邊偵查任務切換閾值（Cowork 剪貼超過兩輪切 Code handoff）
- **§5.2**：Git 狀態 SSoT 規則（`--force-with-lease` 被擋的強制 SOP）

兩節的共同特徵：**跨三窗的操作流程**（Cowork 與 Code 之間、多終端機視窗之間）。不像 §1-§4（權責邊界、源頭事實、Handoff ADR 欄位、長度管制）是**原則性規範**。

架構圖作者（Cowork 2026-04-24）認為這類「具體 how-to」值得跟「原則 why-to」分開標示，拉 §5 為 L3 獨立層。但沒有同步**拆檔案**，造成 L2/L3 定位的邏輯模糊。

### 三個選項的考量背景

Cowork 產出架構圖時已提出三選項：

- **選項 A · 合併**：L3 併回 L2，治理架構從五層改四層
- **選項 B · 拆分**：L2 拆成「原則篇」（留 L2）+「操作篇」（升到 L3 獨立文件）
- **選項 C · 維持現狀 + 寫作規則**：L2/L3 都在 WE 裡，但定義清楚「L3 只收跨三窗的操作流程，其他留 L2」

本 ADR **採納選項 C**，並透過條文明確寫作規則讓未來新規則能被清楚分類，同時保留觸發重構的條件（當實戰證實維持現狀不夠時）。

### 上位法依據

- **憲法 v0.2 第四條 記憶層次**：事件歷史層、實施細則層等各有職能。L2 vs L3 本質上是實施細則層內部的「原則 vs 操作」分工，本 ADR 釐清這個分工的寫作規則。
- **憲法 v0.2 第四條補款 同層原子化**：既然 L2 / L3 是同一個檔案 `working-environment.md`，同層原子化規則本來就適用——規則散在 §1 和 §5 但講同件事是違憲。

### 與 H5 的關係（否決選項 A 的關鍵）

H5 ADR 第一條規定「≥ 軟上限 × 1.15 = 230 行」觸發拆分 ADR。`working-environment.md` 目前已達 629 行（歷史記載，二手），選項 A（合併）如果成立，會讓 WE 仍維持在 629 行以上。但**選項 A 的合併是概念合併不是檔案合併**——架構圖改成四層，檔案行數不變——嚴格說不會直接踩 H5 觸發線。

但**選項 A 的合併**若進一步要求「把 §5 和 §1-§4 在概念上徹底整合重寫」，會需要動 §1-§9 的結構，重寫過程中 WE 行數很可能**繼續膨脹**越過 230 行以上（若整合內容變多）。因此選項 A 的潛在副作用仍涉及 H5 的觸發線。

本 ADR 第一條第 3 項「維持現狀」明示「不重構 WE，讓 WE 行數不因本 ADR 變動」，規避 H5 觸發線。

## Decision

### 第一條 · 三選項比較 + 採納選項 C

本 ADR 在三選項中採納**選項 C · 維持現狀 + 寫作規則**。三選項對比如下：

| 選項 | 描述 | 工程成本 | 邏輯清晰度 | 是否踩 H5 | 採納？ |
|---|---|---|---|---|---|
| **A · 合併** | L3 併回 L2，架構從五層改四層 | 低（只改架構圖） | 🟡 架構圖乾淨，但 §5 失去視覺權重 | 🔴 潛在踩線（若觸發 WE 結構重寫） | ❌ |
| **B · 拆分** | L2 拆成「原則篇」（留 L2）+「操作篇」（升 L3 獨立文件） | 高（實質文件重構、跨引用更新） | ✅ 邏輯乾淨 | ✅ 不踩（WE 反而瘦身） | ❌ |
| **C · 維持現狀 + 寫作規則** | L2/L3 都在 WE 裡，定義清楚「L3 只收跨三窗的操作流程」 | 極低（只加寫作規則） | 🟡 分層概念仍模糊，但操作層清楚 | ✅ 不踩 | **✅ 採納** |

**選項 A 否決理由**：
1. 架構概念合併不消除 WE 文件內部分工問題——§5 仍在 §1-§4 之後，仍是 L2 檔案的一部分
2. 若進一步要「架構合併 + 文件重寫」則工程成本上升，且重寫過程 WE 行數可能越過 H5 觸發線（見 Context §H5 關係）
3. 治理架構圖變四層後，L3 這個概念消失，但 §5 的「跨三窗操作 SOP」本身的特殊性質無處安放

**選項 B 否決理由**：
1. 拆分重構成本遠大於痛點——目前 L2/L3 邊界模糊只在架構圖理解上造成輕微混淆，不影響實際操作
2. 需要大量跨引用更新：憲法、速記卡、CLAUDE.md、skill 的任何引用 WE §5 的位置都要改
3. 物理拆成兩份檔案後，跨 session 載入成本增加（Cowork 開場要讀兩個檔案）
4. 業界經驗（ADR Lightweight 原則）建議「拆分成本 > 痛點成本時，不拆」，本案屬此

**選項 C 採納理由**：
1. 成本最低（只加寫作規則，不動現有檔案結構）
2. 不踩 H5 觸發線
3. 寫作規則能讓**未來**新規則有明確分類依據，止住邊界模糊持續擴大
4. 保留第四條的觸發條件，若未來證實不夠，可升級到選項 A/B

**明確放棄**：選項 A 和 B 不是「永遠放棄」，而是「當下成本不划算」。第四條的重新評估觸發條件若滿足，可重啟討論。

### 第二條 · L2 / L3 定位差異

以下為 `working-environment.md` 內部章節的寫作規則：

| 層 | 內容類型 | 對應章節 | 範例 |
|---|---|---|---|
| **L2**（實施細則層）| 為什麼 + 原則性怎麼做 | `working-environment.md` §1-§4、§6-§9 | §1 職責邊界、§2 源頭事實清單、§3 Handoff ADR 欄位、§4 長度管制、§6 外部共識對照、§7 落地動作、§8 Exit Gate |
| **L3**（操作 SOP 層）| 跨三窗的具體步驟流程 | `working-environment.md` §5 | §5.1 多邊偵查切換閾值、§5.2 Git 狀態 SSoT 規則 |

**L2 的判斷特徵**：
- 寫「為什麼要這樣做」（論述層）
- 寫「這類事的原則性處理」（規範層）
- 不給具體指令序列

**L3 的判斷特徵**：
- 寫「遇到 X 情境時依序跑什麼指令」（流程層）
- 跨視窗（Chat / Cowork / Code）至少兩個
- 可執行步驟，不是原則

### 第三條 · 寫作判斷規則（三題測試）

新規則要寫進 `working-environment.md` 時，作者跑下列三題測試，決定該進 §1-§4/§6-§9（L2）還是 §5（L3）：

| 題號 | 問題 | YES 數量意義 |
|---|---|---|
| Q1 | 此規則涉及 Chat / Cowork / Code 至少兩個視窗交接嗎？ | 涉及跨窗協作則傾向 L3 |
| Q2 | 此規則的核心是可執行步驟流程（不是原則論述）嗎？ | 是步驟流程則傾向 L3 |
| Q3 | 此規則離開這條後下游會壞嗎？（不是最佳實踐，而是缺了 build 就斷） | 是強制性步驟則傾向 L3 |

**判定規則**：

- 三題皆 YES → **L3**（寫進 §5 或新增 §5.X 子節）
- 三題皆 NO → **L2**（寫進 §1-§4/§6-§9 對應主題的章節）
- 混合（1-2 個 YES）→ **預設 L2**（寬進嚴出）

**「寬進嚴出」的理由**：
- 多數規則是「原則 + 少量步驟」混合體，預設 L2 能避免 §5 肥大化
- §5 一旦過肥，WE 整體就會踩 H5 觸發線
- 若某規則進 L2 後實戰證明更適合 L3，可在下次 WE rev 時遷移——遷移比「一次到位」風險低

### 第四條 · 重新評估觸發條件

本 ADR 採選項 C 是治標，不是終局方案。以下任一成立，**必須**重新評估是否升級到選項 A 或 B：

1. **WE §5 超過 H5 軟上限 × 1.15（= 230 行）**：§5 本身膨脹到觸發 H5 拆分 ADR 時，代表 L3 內容體量已不適合寄生在 L2 檔案，必須重新考慮選項 B（拆分）
2. **出現第三條跨三窗協作 SOP**：當 §5 要加第 5.3 節時（目前只有 §5.1、§5.2），代表「跨三窗操作 SOP」已成體量，選項 A（合併）的架構乾淨性優勢被稀釋，應評估選項 B
3. **新 SOP 與 §5 既有條文衝突**：若未來某個跨窗 SOP 的規則與 §5.1 / §5.2 的規則矛盾，代表 §5 內部已需要進一步分類，可能需要 §5 本身拆成多節或獨立文件
4. **WE 整體行數逼近 H5 硬上限前警戒線**：若 WE 行數 ≥ 720 行（WE 硬上限 900 × 0.8），則不論 §5 情況如何，都要啟動 H5 ADR 的拆分機制——屆時本 ADR 可能同時被觸發重新評估

**重新評估的動作**：

任一觸發條件成立時，Cowork 一週內起草新 ADR（`adr-l3-positioning-rev2-YYYY-MM-DD.md`），重新評估選項 A/B/C，不沿用本 ADR 的 C 採納。

### 第五條 · 架構圖呈現調整

2026-04-24 Cowork artifact `paulkuo-governance-architecture-2026-04-24` 的 L3 呈現需**明文調整**：

- 原呈現：L3 作為獨立層，與 L2（WE §1-§9）並列
- 調整後：**L3 明確標註「L3 是 L2 功能子集（§5），不是獨立檔案」**

具體調整方式（由下次 Cowork 執行或 artifact 維護者同步）：

1. 架構圖 L3 框內加註記：「⚠️ L3 物理上是 L2（`working-environment.md` §5）的子章節，非獨立檔案」
2. L2 / L3 之間加視覺連接（虛線或箭頭），表明從屬關係
3. 若 artifact 未來 port 為 paulkuo.tw Astro 頁面（見 `worklog-2026-04-24.md` 待辦），port 前先修正架構圖的 L3 呈現

### 第六條 · 寫入位置

本 ADR 是規則源頭。`working-environment.md` 開頭新增「L2/L3 寫作規則」導引段落（1-3 行），指向本 ADR 全文：

```markdown
## 0.1 L2/L3 章節寫作規則

本檔 §5 為 L3（操作 SOP），§1-§4 / §6-§9 為 L2（實施細則）。
新規則入檔前跑 `adr-l3-positioning-2026-04-25.md` §三的三題測試判斷章節歸屬。
```

本 ADR **不**直接改 WE（避免單一 commit 混改規範層與被規範檔）。實際 WE 編輯由下次 Cowork 在獨立 commit 完成。

## Consequences

### 正面影響

- **L2/L3 邊界有成文規則**：未來新規則不再依賴作者個人判斷，有三題測試保底。
- **工程成本最低**：不動現有檔案，不動跨引用，Cowork / Code 的閱讀負擔不變。
- **保留升級路徑**：第四條觸發條件讓選項 A/B 仍在備援位，當實戰證明選項 C 不夠時可無負擔升級。
- **避免撞 H5 觸發線**：選項 A/B 各自的行數風險（見 Context / §一）都被規避。

### 負面影響

- **分層概念仍有模糊感**：讀者看架構圖會覺得 L3 像獨立層，但實際上是 L2 的子節。第五條的架構圖註記是緩解，不是根治。
- **寫作規則依賴作者自覺**：三題測試是人工流程，沒有 lint 檢查機制（H7 無法以 bash 自動判斷「這條規則屬 L2 還 L3」）。可能出現規則寫錯位置，只能靠 code review 事後糾偏。
- **未來新 SOP 的累積壓力**：§5 每新加一節，本 ADR 的穩定性就降低一分。第四條條件 2（§5.3 出現）是明示的早期警告線，但實戰中可能因其他 SOP 壓力被忽略。

### 中性觀察

- **本 ADR 不禁止新操作 SOP 寫進 WE**：反過來說，§5 可以繼續長大，直到觸發第四條。換言之本 ADR 是「觀察期方案」，觀察三題測試能不能維持實戰秩序。
- **本 ADR 不涉及 L1 / L4 / L5 的重構**：架構圖其他層未動，本 ADR 只針對 L2/L3 分層。L1（憲法）、L4（skill）、L5（infra）的歸屬清楚，不需釐清。
- **本 ADR 不動 cloud 層 / repo 層 / cache 層命名**：H6 ADR 已處理 skill 儲存層命名衝突，本 ADR 不重複此議題。

## Appendix

### 附錄 A · 三題測試範例

以下為 2026-04-25 時已存在或候選的治理規則，套用三題測試的分類結果：

**範例 1 · §5.1 多邊偵查切換閾值**

- Q1 涉及多視窗？**YES**（Cowork + Code + Paul 終端機）
- Q2 可執行步驟？**YES**（「超兩輪剪貼 → 停 → 寫 Code handoff → 寫 PENDING」明確步驟）
- Q3 離開會壞？**YES**（不遵守則多邊偵查失效、錯誤率升）

→ 三題皆 YES，分類 **L3**（寫進 §5）。實際現況也是在 §5.1。✅ 判斷正確

**範例 2 · §2 源頭事實清單規範**

- Q1 涉及多視窗？YES（Chat / Cowork / Code 皆要附 F-ID）
- Q2 可執行步驟？**NO**（是論述 + 規範格式，不是步驟序列）
- Q3 離開會壞？YES（不附 F-ID 會重演 1085 vs 522 踩坑）

→ 2 YES + 1 NO = 混合，預設 **L2**。實際現況在 §2（L2）。✅ 判斷正確

**範例 3 · §4.3 CLAUDE.md 判斷樹**

- Q1 涉及多視窗？**NO**（只規範作者寫作時的前置判斷）
- Q2 可執行步驟？YES（三問判斷樹）
- Q3 離開會壞？**NO**（不跑判斷樹只會讓 CLAUDE.md 容易膨脹，不是 build 斷）

→ 1 YES + 2 NO = 混合，預設 **L2**。實際現況在 §4.3（L2）。✅ 判斷正確

**範例 4 · 假設性：「Cowork 開場必跑的檢查清單」**

- Q1 涉及多視窗？YES（Cowork 開場要讀 Code 產出的 worklog）
- Q2 可執行步驟？YES（逐條 checklist）
- Q3 離開會壞？YES（不跑會漏接 Code 最新狀態）

→ 三題皆 YES，分類 **L3**（未來應寫進 §5.3，若新增）。但會觸發本 ADR 第四條條件 2（§5.3 出現），屆時重新評估選項 A/B。

### 附錄 B · 為什麼不選「L3 完全廢除概念」

有第四選項「L3 概念完全廢除，跨三窗 SOP 直接寫進憲法」，本 ADR 不採納因為：

1. 憲法 v0.2 是上位法，寫具體操作會污染憲法層級（憲法第五條「記憶擴充」的反向推論——上位法不含下位操作）
2. 憲法若要含操作 SOP，需要修憲（v0.3 以上），成本遠高於保留 L3 概念
3. 即使長期目標是縮小 L3，現階段直接跳過 L3 會讓 §5.1 / §5.2 這兩節無處安放

因此選項 D 在可選範圍外，本 ADR 正文不列入比較表。
