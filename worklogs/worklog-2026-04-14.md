- **Stitch MCP 寫入路徑 `InvalidArgument`**（未解）：`create_design_system`（全域 + 帶 projectId 兩種）與 `update_design_system`（傳假 asset name 試 upsert）皆失敗。嘗試過的 payload 形狀：`{colorMode,font,roundness,saturation,customColor}` 平坦 / 包 `designSystem` 頂層 / 包 `{displayName, theme}` 巢狀（對齊既有 asset 的 shape）全都回相同錯誤。讀取端（list_projects / list_design_systems / get_screen）完全正常，可見不是連線/驗證問題，是 MCP server 這組 write endpoint 的 schema 與 list 回傳不一致。→ 暫解：design system 落地到 `docs/design-system.md`；待修解：Stitch MCP 版本更新或改走 Stitch Web UI
- 測試用空殼專案 `projects/6228487101449773171` 留在 Stitch 帳號下（Stitch MCP 無 delete_project），需要 Paul 有空在 Stitch 網頁 UI 手動刪，否則會佔位
- Token gate 本機無法驗證（CORS：`access-control-allow-origin` 只允許 `https://paulkuo.tw`，localhost 被擋）。這是既有行為，非本次改動造成。需線上 deploy 後再跑 Section 5.4 / 6.2 驗證。
- Paul 之前在終端機設定 Stitch MCP 沒生效的真正原因：`~/.claude.json` 的 user-scope `mcpServers` 是空的，推測他跑 `claude mcp add` 時沒加 `--scope user`，被寫到 project/local scope 所以只對單一 repo 有效。Cowork 根本沒看到。→ 已透過 helper script 一次補齊
- osascript 跑指令內含特殊字元（!、$ 等）時 bash quoting 很容易爆。用 `do shell script` 送 bash 時建議「指令越短越好」，複雜邏輯包進 script 檔案再呼叫

- Claude Desktop / Cowork 對於不支援的 MCP entry 格式（如 type:http）會「靜默跳過」，連 failed 狀態都不顯示。診斷方式：在 Settings → Developer 看設定的 MCP 是否「整條消失」
