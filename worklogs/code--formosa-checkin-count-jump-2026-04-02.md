## 可能原因（依可能性排序）

### 1. Auto-retry 重複計數（最可能）
PR #91 在前端加了 503/network error 自動重試。如果第一次請求其實成功了（Worker 回 202 因為 KV 寫入慢），前端誤判為失敗而重試，就會多寫一筆 checkin。

**修法**：
- Worker 端加冪等性保護：同一 user_id + 同一分鐘內的重複 POST 直接回傳現有 count，不新增
- 或前端 retry 前先 GET 最新 count，確認是否需要重試

### 2. localStorage 和 API 不一致
頁面載入時讀 localStorage 舊值，打卡後 API 回傳新值但沒同步寫回 localStorage。

**修法**：
- 打卡成功後立即用 API response 的 count 覆寫 localStorage
- 頁面載入時以 API 為準，localStorage 只當離線 fallback

### 3. 連續快速點擊
用戶快速連點打卡按鈕，多次 POST 都成功。

**修法**：
- 前端打卡按鈕點擊後 disable 3 秒
- Worker 端做 debounce（同 user 5 秒內只接受一次）

## 具體修復步驟

1. **開分支**：`fix/checkin-count-jump`
2. **Worker 冪等性**：checkin handler 加入 `user_id + timestamp 分鐘級` 去重
3. **前端 debounce**：打卡按鈕點擊後 disable + loading 狀態
4. **前端 retry 修正**：收到 202 不應觸發 retry（202 = 已接受）
