夜11時半、paulkuo.tw のトップページを開いて、トラフィック分析セクションを確認した。

訪問者：0。

おかしい。今日は記事を読んでくれた人がいたし、翻訳ツールも利用者がいた。昨日のデータでは数百人が訪問していたのに、今日いきなりゼロになるはずがない。Cloudflare Dashboard を開いてみると——Page views 193、Visits 130。

ゼロと130は誤差ではない。まったく異なる二つの世界だ。「誰も来なかった」のか、「130人以上が訪れた」のか。どちらが正しいのか、それとも両方間違いなのか？

## Cloudflare の二つの世界

サイトを Cloudflare に載せているなら（世界中で4,100万以上のサイトがそうだ）、Cloudflare に実は二つのまったく異なる分析システムがあることを知らないかもしれない。

**第一のシステム：Zone Analytics（HTTP Traffic）。** CDN層のデータ。Cloudflare ネットワークを通過するすべてのHTTPリクエストが記録される。総リクエスト数、帯域幅、国別分布、ユニーク訪問者数（IPで重複排除）がわかる。正確で、Googleのクローラー、ChatGPTのcrawler、各種監視botも含めて、すべてをカウントする。

**第二のシステム：Web Analytics（RUM beacon）。** ブラウザ層のデータ。Cloudflare がページにJSビーコンを注入し、実際の人間がブラウザでページを読み込んだときにのみ発火する。Botは JavaScript を実行しないため、非人間トラフィックを自然にフィルタリングする。

流量分析アーキテクチャを設計する際、一度の反復を経て、Web Analytics の GraphQL API を主要データソースに選んだ。理由は合理的に聞こえた——人間だけをカウントするので、より正確だと。最初は Zone Analytics を使っていたが、botを含むため数字が膨らんでいた。サイトを設定したばかりなのに、一日で千以上のVisitが表示されていた。

ロジックは問題なかった。しかし一つの過ちを犯した：「このAPIがどんなデータフィールドを提供できるか」は検証したが、Web Analytics（RUM beacon）が「返す数字が正確かどうか」は検証しなかった。

## Adaptive sampling：130を0にする魔法

調査の結果、根本原因を見つけた。Cloudflare の Web Analytics GraphQL API は、Adaptive Bit Rate（ABR）と呼ばれるサンプリング技術を使っている。

原理はシンプルだ：Cloudflare は毎秒7億件以上のイベントを処理している。すべてのクエリで全生データをスキャンしていたら、システムコストが膨大になる。そこでデータを複数の解像度——100%、10%、1%——で保存し、クエリ時にデータ量と複雑さに応じて自動的に解像度を選択して結果を返す。

高トラフィックサイトではまったく問題ない。日間訪問者が10万なら、10%サンプリングでも1万件のデータポイントがあり、統計的に十分な精度がある。

しかし低トラフィックサイトではどうか？paulkuo.tw は一日約130人の実際の人間の訪問者。APIサンプリング後、返されるvisitsはすべて100の倍数——0、100、200。130は切り捨てられて0になった。

過去30日分のAPIデータを確認したところ、visits = 0の日が数日あり、すべての値が100の倍数だった。これは「今日壊れた」のではなく、初日からこういうロジックだった。今日初めてAPIデータとDashboardを比較しただけだ。

Cloudflare のドキュメントを確認したところ、彼らはクエリ結果がどの程度正確かをユーザーが検証する方法が現時点ではないと認めている。つまり、数字を受け取るが、その誤差範囲を教えてくれる人はいない。

## さらに大きな数字がある

Web Analytics のサンプリング問題を整理した後、Zone Analytics を見てみると——ユニーク訪問者数は1,100だった。

Web Analytics Dashboard は130と言い、Zone Analytics は1,100と言い、APIは0と言った。

3つの数字、同じ日、同じウェブサイト。

1,100と130の差——約千の「余分な訪問者」はbotだった。私のサイトは最初からAIフレンドリーに設計されており、llms.txt、JSON-LD、MCPサポートがすべてAIシステム向けに構築されている。だからGPTBot、ClaudeBot、Bingbot などのクローラーが勤勉にコンテンツを取得している。Zone Analytics はすべてのIPを忠実に記録する——人間か機械かを問わず。

Imperva の2025年レポートによると、自動化トラフィックは2024年に初めて人間の活動を上回り、グローバルインターネットトラフィックの51%を占めた。悪意のあるbotは37%を占めた。Cloudflare の2025年年次レビューでも、AI botのHTMLページリクエストが4.2%、Googlebot単体で4.5%を占めていた。

私のサイトのユニークIPの88%がbot、12%が実際の人間。極端に聞こえるが、統計的にはまったく合理的だ。今年1月から、「Make something Agent Want」が技術コミュニティのコンセンサスになっていることを明確に感じるようになった。これが、サイトの人間Visitだけを見る執着を再検討するきっかけになった。

## 間違えた三つの道

現在のソリューションに至る前に、三つの過ちを犯した。

**第一の過ち：計画段階で技術偵察をしなかった。** Web Analytics RUM を主要データソースに選ぶ前に、adaptive sampling の動作制限を調査せず、APIの返り値とDashboardを交差検証しなかった。当時5分かけてこれをしていれば、問題全体が計画段階で発見されていた。

**第二の過ち：問題発見後に急いで修正した。** 「Zone-level APIのサンプリング粒度のほうが良いはず」と直感的に判断し、「人間中心」の執着を持ったまま修正版をGitHubにプッシュした。プッシュ後にCloudflareコミュニティでzone-level APIにも同じvisits=0の問題が報告されていることを発見。revertした。行ったり来たりで時間を無駄にし、git historyを汚してしまった。

**第三の過ち：「botなし」と「正確」を同一視した。** Web Analytics は確かに人間だけをカウントするが、「botなし」は「数字が正しい」を意味しない。サンプリング精度とbotフィルタリングはまったく独立した二つの事柄だ。私はそれらを混同していた。

AI Agentとの工程原則は「先に偵察、それから行動」だが、AIも見落とす（おかしいな、skillにはもう書いてあるのに）、私も見落とす。問題を発見したとき、急いで修正に走り、やるべき調査を飛ばしてしまう誘惑は特に強い。

## 完全な観客プロフィールを見る

最初はbotトラフィックをノイズとして排除しようとしていた。しかし paulkuo.tw には [AI-Ready なアーキテクチャ設計](/articles/ai-ready-continuous-optimization)がある——llms.txt がAIシステムにサイト構造を理解させ、JSON-LD が構造化知識を提供し、MCPプロトコルがAI agentの直接操作を可能にする。だから、AI botにコンテンツを読ませることはノイズではなく、影響力の一部でもある。

正しい問いは「botをどう排除するか」ではなく、「完全な『読者の観客席』をどう見るか」に変わった。

二つの独立した指標が必要だった：

- **人間の訪問者**：何人の人間の読者が記事やツールを見たか——これは旧世界のコミュニティ影響力を測る核心指標
- **AI/Bot訪問者**：何のAIシステムがコンテンツを読んでいるか——これはAI-Ready戦略の成果指標

最終的なアーキテクチャも複雑ではない。人間の訪問者には自前のvisit beaconを使用——ページ読み込み時にCloudflare WorkerにPOSTを送信し、WorkerがIP + User-Agentの匿名ハッシュで日次重複排除を行う。サイト全体の訪問者にはZone Analyticsの正確なIP重複排除を継続使用。二つを引き算すればAI/Botの量になる。

3つの数字すべてが自前で計算され、Cloudflareの推定に依存しない。Google Analytics、Plausible、Umamiなどのウェブサイトトラフィック分析ツールと同じ原理だ。ページにトラッキングコードを埋め込み、自分で一人一人を数える。違いは、サードパーティツールをインストールする必要がなく、既存のサーバーインフラ上で直接動作することだけだ。

## あなたのデータは誤解を招いているかもしれない

2025年、自動化トラフィックが初めて人間の活動を上回り、グローバルインターネットトラフィックの51%を占めた。AI crawlerの取得量は同年15倍以上に増加した。あなたのサイトは人間だけでなく、機械にも読まれている。

Cloudflare Web Analytics のフリープランを使っていて、日間訪問者が数百以下なら、ダッシュボードのvisits数字は、私と同じく、サンプリング後の推定値であり正確な値ではない可能性が高い。

これはCloudflareが悪いという意味ではない。CDN、DNS、セキュリティ保護は業界トップクラスだ。Web Analytics の Dashboard UIの数字は正確だ。しかし、APIでデータを自分のダッシュボードに引き出す場合、低トラフィックシナリオでは自分で検証する必要がある。APIの数字とDashboardを比較すれば、5分で答えがわかる。

データの乖離が大きすぎる場合、サイト運営者はより直接的な方法に立ち返り、自前で正確な訪問カウントメカニズムを構築する必要があるかもしれない。これはCloudflareが信頼できないからではなく、トラフィック規模がまだ小さいときにサンプリングが本質的に不正確になりやすいからだ。オーディエンスが拡大するにつれ、サンプリング精度は自然と向上する。しかし、トラフィックがまだ成長段階にあるときこそ、正確な数字が最も不可欠だ——方向を判断し戦略を調整するための重要な時期だからだ。

サイト運営は感覚だけに頼ることはできない。データは健康診断レポートのようなものだ。最も基本的なリーチ数すら正確に把握できなければ、自分の体重、体脂肪率、生理指標をまったく知らないようなもので、改善と成長は明確で根拠のある判断の上に築くことが難しくなる。
