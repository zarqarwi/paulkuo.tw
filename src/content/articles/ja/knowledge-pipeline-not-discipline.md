3月の最初の2週間で Get筆記に保存したものを数えてみました。200近くのアイテムです。記事、podcast の概要、得到 App のコース・ノート、会議の音声議事録です。保存するときはいつも「後でこれを見る必要がある」と思っています。

その後？何もありません。

アプリを開くと、すべてが同じタイムラインに詰め込まれています。コース・ノートと買い物リストが混在しており、3週間前に保存した AI Agent アーキテクチャに関する深い記事は、後で保存した20編の記事に埋もれています。

あなたはこのシーンに見覚えがあるはずです。私の周りの友人たちも同じ悩みを抱えています：きれいなデータベースを構築し、3日間書いた後は二度とメンテナンスしませんでした。他のツールを使っている人も大変です。どのプラグインを使うか、タグをどう設計するかで悩んでいるうちに、整理の気力が尽きてしまいます。

問題は明らかです。収集が難しいのではなく、整理が難しいということです。整理が難しい理由は、私たちそれを「自律」で維持する必要がある習慣として扱っているからです。

しかし自律は世界で最も信頼できないリソースです。

## 考え方を変える：整理に人間の参加が不要だったら？

[1人と1つの AI で完成できる工事量](/articles/super-individual-case-study)という考え方から得た気づきの1つは：人間の継続的な手動介入が必要なプロセスはすべて崩壊するということです。あなたの自律性が足りないからではなく、手動プロセスの維持コストはデータ量に比例して線形に増加しますが、私たちの注意力は指数関数的には成長しないからです。

ソフトウェアエンジニアリングに pipeline という概念があります——データが A から B に流れ、B から C に流れ、各段階が次の段階を自動的にトリガーし、人間が側で見張る必要がありません。CI/CD は pipeline で、ETL は pipeline で、さらにはあなたの家の食器洗い機も一種の pipeline です。汚い皿が入り、きれいな皿が出てきます。あなたは側で1枚ずつ洗う必要がありません。

私がしたかったことは非常にシンプルです：知識管理も pipeline に変える。可能かどうか、わかりません。まず試してみましょう。

## 4段階：収集 → 同期 → 分類 → 利用

パイプライン全体を分解すると、4つの段階があります。（全体像を把握するには、この[インタラクティブなフローチャート](/knowledge-pipeline-flowchart-ja.html)をご覧ください。）

**収集**はモバイル端で完了します。Get筆記は統一入口です：良い記事を見つけたら保存。podcast で心に残った部分があれば保存。得到 App のコース・ノートは自動的に同期されます。別の操作は不要です。会議の音声をアップロードすると、AI が自動的に文字起こしと要約を行います。この層の重要なポイントは「統一入口が1つだけ」ということです。すべてが Get筆記に入り、分散しません。

**ローカルに同期**するには Python スクリプトを使います。Get筆記に OpenAPI があるので、`sync_notes.py` を書きました。crontab で毎晩23時に自動実行するよう設定しました。新しく追加されたノートだけをプル（増分同期）し、Markdown 形式に変換してローカルの `notes/` フォルダに保存します。毎朝起きると、前日保存したものはすでにコンピュータの中に静かに横たわっています。

**自動分類**は、パイプライン全体の中で最も心を込めた部分です——そして最も誇りに思っている部分です。

**利用**では、分類されたノートは全文検索で直接検索でき、また Cowork Skill を通じて自然言語で検索できます。「あのロブスターについての記事を見つけて」と言うと、API で検索して結果を返します。ファイル名を覚える必要もなく、どのフォルダに保存されているか覚える必要もありません。

## 3層分類エンジン：各ノートを自動的に「自分の場所」に見つけさせる

分類エンジンは3層の fallback アーキテクチャで、各ノートは上から順に実行され、最初の層がヒットすると以下には進みません。

最初の層は**音声カード検出**です。Get筆記の音声ノートには「音声カード・ノート」というタグが付いています。スクリプトはこのタグを検出すると、会議録音フォルダに分類します。内部はキーワードで8つのプロジェクト・サブフォルダに分かれています——SDTI 関連、CircleFlow 関連、投資家会議関連など、それぞれ独自の場所に。

2番目の層は**コースシリーズ検出**で、最も満足のいく設計です。得到 App のコース記事では、URL に `courseArticleId` パラメータが隠れています。同じコースのすべての記事は同じ courseArticleId を共有します。スクリプトはこの ID を解析し、`_course_registry.json` という動的レジストリと照合します。

このレジストリのすばらしいところは、自動的に拡張されるということです。スクリプトが見たことのない courseArticleId に出会うと、どうしたらいいかわからなくなるのではなく、新しいフォルダを自動的に作成し、このコースをレジストリに登録し、アーカイブを開始します。次回同じコースの別の記事に出会うと、どこに送るか知っています。

新しいコースを開くたびにコードを変更する必要はありません。システムは自動的に新しいコースを認識します。

3番目の層は**キーワード分類**で、最も素朴ですが最も堅牢なフォールバック戦略です。スクリプトはノートのタイトルとテキストの最初の300文字をスキャンし、キーワード・ライブラリと照合して、対応するトピック・フォルダに分類します：AI とテクノロジー、医療と健康、投資と金融、個人の成長、生活の雑記......。どのカテゴリにも分類されないものは「その他」に入り、少なくとも虚空に消えません。

3層の優先順位は重要です：音声カードは最も確実（明確なタグ）で、コースシリーズは次点（構造化 ID）で、キーワードはあいまいマッチですが最も広い対象範囲です。各ノートには必ず居場所があります。

## なぜ手動タグ付けを使わないのか？

パイプラインを構築する過程で、何度も手動タグを設定する方法を試しました。結論は：うまくいきません。

技術的にはうまくいかないのではなく、人間の本性上うまくいきません。記事を保存するとき、あなたの頭の中にあるのは「これはとても役に立つ」です。「これはどのタグシステムの何層に分類すべきか」ではありません。収集の時点で分類の決定を下すよう求めることは、最も不適切な時期に最も認知負荷の高い操作を求めることと同じです。

2番目の問題はタグが漂流することです。1月に設定したタグシステムは、3月には間違っていると感じますが、最初の2ヶ月の数百のノートを遡って再タグ付けすることは不可能です。タグシステムの維持コストはコンテンツ量に比例し、さらに遡及的です——規則を1回変更すると、すべての履歴データを再処理する必要があります。

自動分類の利点は：ルールが変わったら、スクリプトを1回実行するだけです。100個でも1万個でもコストは同じです。

## 「整理」から「利用」へ

パイプラインが構築された後、実は私の仕事のやり方を変えたのは「整理」が速くなったことではなく、「利用」も友好的になったことです。Claude の窓の中で直接データを引き出して、必要な内容を出力できます。

以前は保存したものは保存しないのと同じでした。見つからないからです。今は Cowork の中で直接「最近循環経済についての記事を保存しましたか」と言うことができます。Skill は API で検索し、一致するノートを表示し、概要も付けます。「万維鋼のコースまでやったか確認して」と言うと、`_series_meta.json` の進捗インデックスを読みます。

知識管理の目標は「保存することがいい」ではなく、「使える」ことです。パイプラインが解決するのは整理の問題だけでなく、収集と利用の間の大きな溝を埋めることです。

これは [paulkuo.tw で AI-Ready 継続最適化](/articles/ai-ready-continuous-optimization)をするときのロジックと同じです：人間がシステムに適応するのではなく、システムが人間に適応するようにします。ウェブサイト最適化は自動ループに任せ、知識整理は自動パイプラインに任せます。人間の努力は真の判断力が必要なことに残します。

## 1本のパイプライン、1つの態度

振り返ってみると、このパイプラインは技術的には複雑ではありません。1つの Python スクリプト、1つの crontab スケジューリング、1つの JSON レジストリ、1つの Cowork Skill。機械学習なし。ベクトル・データベースなし。何か奥深い NLP なし。

しかし、それは私を悩ませていた1つの問題を解決しました。

以前は知識管理の記事を見るたびに、「第二の脳を構築する」というナレーティブに引き寄せられ、週末を使って Notion テンプレートを構築し、タグシステムを設計し、使用ガイドラインを書きました。その後2週間で元に戻ります。ツールが悪いのではなく、その方法は本質的に人間の本性に賭けています：毎日手動で整理する力があると賭けています。でも毎日十分な集中力を持つことはできません。

パイプライン思考は人間の本性に賭けません。API の安定性、cron の正確性、プログラムロジックの正確性に賭けます。これら3つのことの信頼性は、どんな人間の自律性よりも高いです。

あなたも知識管理に不安を感じているなら、私の助言はより良いアプリやより美しいテンプレートを探すことではありません。AI と協力して、従来の SOP を手放して、自分自身に1つ質問してください：このプロセスの中で、実は私を必要としない段階はどれですか？どの段階が委譲可能ですか？必要ない段階をパイプラインと AI に任せます。読むこと、考えること、つなぐこと、創造することを自分のために残してください。

私は思います。それが知識管理が持つべき姿です。もちろん、これは私個人のバージョンです。あなたは自分のバージョンを発展させることができます。
