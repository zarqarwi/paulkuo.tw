# Formosa ESG Feedback 全自動修復 Pipeline

這個 skill 負責從 feedback API 撈取用戶回報，AI 分類後自動開 GitHub Issue、產出 Code handoff、更新 feedback 狀態。

## 重要：網路存取方式

沙盒環境無法直接發 HTTP 請求。所有 API 呼叫必須透過 osascript MCP 在 Mac 本機執行 curl。GitHub 操作用 github MCP（不受此限制）。

呼叫範例：
```
mcp__Control_your_Mac__osascript({ script: 'do shell script "curl -s https://example.com"' })
```

---

## Pipeline 流程

### Step 1：撈取新 feedback

用 osascript 執行：
```
do shell script "curl -s 'https://api.paulkuo.tw/api/formosa/feedback' -H 'X-Admin-Token: formosa-admin-2026'"
```

解析 JSON，過濾出 status 為 new、N/A 或 null 的條目。沒有新條目就回報「無新 feedback」結束。

### Step 2：AI 分類

每筆 feedback 判斷三個維度：

**類型**（四選一）：
- Bug — 功能壞掉、資料錯誤、頁面報錯
- UX 建議 — 排版、文案、流程改善
- 環境問題 — 用戶手機/瀏覽器/網路限制，非程式問題
- 已知已修 — 跟已存在的 GitHub Issue 重複

**優先級**：
- P0 — 服務掛掉、全站不可用
- P1 — 核心功能壞（打卡、登入、GPS）
- P2 — 體驗差但不影響核心功能
- P3 — 建議性質

**風險等級**（決定部署流程）：
- Level 0（零風險）：純文案/CSS/i18n。Code push main，自動部署，不需人介入。
- Level 1（低風險）：前端 UI 邏輯。Code push main，Cowork smoke test 驗收。
- Level 2（中風險）：API/CORS/LIFF/前後端互動。PR + Cloudflare preview 驗收。
- Level 3（高風險）：Worker routing、_redirects、D1 schema、auth、KV。PR + Paul 親自驗收。

**活動凍結期**（4/11~4/13）：只接受 P0，其他排到活動後。

### Step 3：處理 Bug 類

對每筆分類為 Bug 的 feedback：

1. 用 mcp__github__create_issue 建 GitHub Issue：
   - owner: zarqarwi, repo: paulkuo.tw
   - title 格式: "Bug: {簡述問題}"
   - body 包含：來源 feedback ID、問題描述、設備資訊、優先級、風險等級、偵察方向（grep 指令）、可能原因、驗證方式
   - 結尾加 "🤖 Auto-generated by Cowork feedback pipeline"

2. P0/P1 Bug 額外用 mcp__github__create_or_update_file 產出 Code handoff：
   - path: worklogs/code--formosa-{描述}-{日期}.md
   - branch: main
   - 內容包含：背景、Step 0 偵察指令、可能原因、具體修復步驟、驗證方式
   - 標明風險等級對應的部署流程

3. 用 osascript PATCH feedback status 為 triaging：
   ```
   do shell script "curl -s -X PATCH 'https://api.paulkuo.tw/api/formosa/feedback/{id}' -H 'X-Admin-Token: formosa-admin-2026' -H 'Content-Type: application/json' -d '{\"status\":\"triaging\",\"admin_note\":\"已開 Issue #{number} 追蹤\"}'"
   ```

### Step 4：處理 UX 建議類

不自動開工單。整理成清單標注「等 Paul 決定」。PATCH status 為 triaging。

### Step 5：處理環境問題類

PATCH status 為 wontfix，admin_note 說明原因（如「瀏覽器限制」）。

### Step 6：Health Check

用 osascript curl 檢查 5 個端點，每個回報 HTTP status code：
- https://mazu.today/projects/formosa-esg-2026/tracker/ 預期 200
- https://mazu.today/projects/formosa-esg-2026/feedback/ 預期 200
- https://mazu.today/projects/formosa-esg-2026/dashboard/ 預期 200
- https://paulkuo.tw/ 預期 200
- https://api.paulkuo.tw/api/formosa/checkin (OPTIONS) 預期 204

### Step 7：產出報告

整理摘要給 Paul：
- 共幾筆新 feedback
- Bug 幾筆（已開哪些 Issue）
- UX 建議幾筆（列出等 Paul 決定的項目）
- 環境問題幾筆（已標記不處理）
- P0/P1 需要緊急處理的特別標出
- Health Check 5 項端點結果

## 注意事項

- 疑似重複的 feedback 合併處理只開一張 Issue
- osascript 裡的 JSON 雙引號要用反斜線跳脫
- Worker deploy 必須帶 --config（Cowork 無法 deploy，只產出指令給 Paul）
- Admin token: formosa-admin-2026
- 品牌名稱（feedback #5）等 Paul 定案才能動
