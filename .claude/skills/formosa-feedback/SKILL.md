# Formosa Feedback Pipeline

用戶回報 → Cowork 分類 → GitHub Issue → Code 修復 → Issue close → feedback log 自動顯示「✅ 已修復」

這是一條完整的閉環管線。每個環節都有明確的規則，漏掉任何一步都會導致回報石沈大海或重複處理。

---

## 管線總覽

```
用戶在 Tracker/LINE 提交回報
       ↓
POST /api/formosa/feedback → D1 feedback table (status: new)
       ↓
Cowork 撈 GET /api/formosa/feedback → 分類 + 分級
       ↓
開 GitHub Issue（必帶 feedback-id: + 風險等級 L0-L3）
       ↓
Handoff 給 Code session 修復
       ↓
Code 修完 → Issue close
       ↓
GitHub Action feedback-auto-close.yml 自動 PATCH feedback → fixed
       ↓
mazu.today/feedback/log/ 即時顯示「✅ 已修復」
```

---

## Step 1：撈取回報

每次 Cowork session 開場、或 Paul 提到有新回報時執行。

```bash
# 從沙盒打 API（如果網路不通，請 Paul 在本機跑）
curl -s "https://api.paulkuo.tw/api/formosa/feedback" \
  -H "X-Admin-Token: {FORMOSA_ADMIN_TOKEN}" | python3 -m json.tool
```

也可以用 Chrome MCP 直接看管理後台的回報頁面。

**比對已知 Issue：** 撈到的回報先跟 GitHub 上已開的 Issue 比對，避免重複開票。
用 GitHub MCP `list_issues` 搜尋 `formosa` label 的 open issues。

---

## Step 2：分類

每筆回報歸入三類之一：

| 類型 | 定義 | 處理方式 |
|------|------|---------|
| **Bug** | 功能壞掉、資料錯誤、頁面空白 | 直接開 Issue，標嚴重程度 |
| **UX 建議** | 排版、文案、流程改善 | 列出讓 Paul 決定，OK 才開 Issue |
| **環境問題** | 用戶手機/網路/瀏覽器限制 | 標記但不處理，admin_note 說明原因 |

分類判斷由 Cowork 負責（技術面）。UI/UX 類的最終決定權在 Paul。

---

## Step 3：風險分級

所有 Issue 一律標 L0-L3，不管來源（使用者回報、Paul 自測、Cowork 診斷都一樣）。
風險等級決定部署流程，Code session 拿到 Issue 就知道該怎麼走。

| 等級 | 適用範圍 | 部署流程 |
|------|---------|---------|
| **L0** | 純文案 typo、CSS 微調、i18n 翻譯 | Code push main → CI 自動部署 → smoke test 自動驗 |
| **L1** | 前端 UI 邏輯、元件狀態、顯示不同步 | Code push main → CI 部署 → Cowork 跑 smoke test |
| **L2** | API endpoint、CORS、LIFF、前後端互動 | Code 開分支 → PR → Cloudflare preview → 驗收 → merge |
| **L3** | Worker routing、_redirects、D1 schema、auth/權限、KV 結構 | PR → preview → **Paul 親自驗收** → merge → Paul 本機 deploy |

**判斷原則：** 壞了只影響某個功能不順 = L1。壞了全站掛 / Worker 500 / 資料不見 = L3。

### 活動凍結期（起駕前 1 天 ~ 活動結束後 1 天）
- 只修 P0（服務掛掉），其他排到活動後
- 低風險門檻拉高（只有純文案/CSS 算 L0）
- 除非服務掛掉否則不部署

## Step 4：開 GitHub Issue

用 GitHub MCP `create_issue` 開票。以下是**強制格式**，每個欄位都有存在的理由。

### Issue 模板

```markdown
## 來源

feedback-id: {ID}
feedback-id: {ID2}

Feedback #{ID}（{日期時間}）
{如有同類回報，列出}

## 問題描述

{用戶原文摘要，保留關鍵細節}

## 設備資訊

- {裝置 / 瀏覽器 / 解析度}

## 分類

- **類型**：Bug / UX 建議 / 環境問題
- **優先級**：P0（服務掛掉）/ P1（核心功能）/ P2（非核心）
- **風險等級**：Level {0-3}（{對應部署流程描述}）

## 偵察方向

（給 Code session 的起手式偵察指令）

## 可能原因

1. {假說 A}
2. {假說 B}

## 驗證方式

1. {怎麼確認修好了}

## ⚠️ {緊急程度提醒，如有}

🤖 Auto-generated by Cowork feedback pipeline
```

### 強制規則（漏掉任何一個都會破壞管線）

1. **`feedback-id: {數字}` 必須存在且格式正確**
   - 放在 `## 來源` 區塊的最前面，每個 ID 一行
   - 不能用 `Feedback #24` 這種自然語言格式（regex 抓不到）
   - 不能包在 HTML comment 裡（某些環境不可靠）
   - 一個 Issue 可以對應多筆 feedback（同一 bug 多人回報）
   - **為什麼：** `feedback-auto-close.yml` 用 `feedback-id:\s*\K\d+` regex 抓 ID，
     Issue close 時自動 PATCH feedback 狀態為 fixed。格式不對 = feedback log 不會更新。
     4/11 事故：Issue #161/#162/#163 都漏了這個欄位，導致回報頁面沒自動同步。

2. **風險等級 L0-L3 必須標明**
   - 在 `## 分類` 區塊明確寫出
   - GitHub label 加上 `L0` / `L1` / `L2` / `L3`
   - **為什麼：** Code session 根據等級決定部署流程。沒標等級 = Code 可能走錯流程。
     4/05 事故：Issue #102 auth bug 沒標等級，Code 直接 push main，應該走 L3 PR+Paul 親驗。

3. **偵察方向要具體**
   - 給 Code 可以直接複製貼上的 grep/curl 指令
   - 不要只寫「檢查相關程式碼」
   - **為什麼：** Code session 是全新 context，具體指令能讓它快速定位問題。

---

## Step 5：交接 Code

如果 Bug 需要 Code 修復，用 session-handoff skill 寫 handoff 文件。
Handoff 裡要帶**模型建議**（Haiku/Sonnet/Opus + Effort 等級）。

口頭跟 Paul 總結時也要講模型建議，不能只寫在文件裡。

---

## Step 6：Issue Close → 自動同步

當 Code 修完、Issue 被 close 時：

1. `feedback-auto-close.yml` 觸發
2. 掃 Issue body 的 `feedback-id:` 欄位
3. 對每個 ID 執行 `PATCH /api/formosa/feedback/{id}` → `{"status":"fixed","admin_note":"Auto-closed: Issue #{number} closed"}`
4. Feedback log 頁面自動顯示「✅ 已修復」

**如果自動同步沒觸發（檢查 Actions tab）：**

手動 PATCH：
```bash
TOKEN="{FORMOSA_ADMIN_TOKEN}"
curl -X PATCH "https://api.paulkuo.tw/api/formosa/feedback/{ID}" \
  -H "Content-Type: application/json" \
  -H "X-Admin-Token: $TOKEN" \
  -d '{"status":"fixed","admin_note":"Issue #{number} closed — {修復摘要}"}'
```

Cowork 沙盒網路可能打不到 api.paulkuo.tw，這時產出 curl 指令讓 Paul 在本機跑。

---

## Step 7：驗證

修復完成後，在 feedback log 頁面驗證狀態是否正確更新：

```
https://mazu.today/projects/formosa-esg-2026/feedback/log/
```

用 Chrome MCP 開頁面，確認：
- 對應的回報出現在「✅ 已修復」區塊
- admin_note 顯示正確的 Issue 編號和修復摘要
- 日期顯示為修復日期

---

## 透明度原則

Paul 的核心要求：**回報 ≠ 修復，但每筆回報都要有回饋。**

- 所有回報公開呈現，不需要等 triage 完才顯示
- 「未處理」不代表「不該顯示」
- 頁面排列邏輯：📥 已收到 → 🔧 處理中 → ✅ 已修復 → 💬 已知 & 說明
- 即使是無法修復的問題（環境限制），也要用 `wontfix` + `admin_note` 說明原因

## API 參考

| 動作 | 方法 | Endpoint | Auth |
|------|------|----------|------|
| 撈所有回報 | GET | `/api/formosa/feedback` | X-Admin-Token |
| 公開狀態（log 頁面用）| GET | `/api/formosa/feedback/status` | 不需要 |
| 更新單筆 | PATCH | `/api/formosa/feedback/{id}` | X-Admin-Token |

### Feedback 狀態機

```
new → triaging → fixing → fixed
                       → wontfix
```

- `resolved_at`：status 改為 `fixed` 時自動填入
- `admin_note`：Cowork 寫的公告文案，會顯示在 log 頁面

---

## 回滾機制

萬一修復反而造成更大問題：

- **前端/Worker**：`git revert` → CI 自動重部署（3-5 分鐘）
- **D1 資料庫**：time-travel 回溯（5-10 分鐘），schema 改動一律 L3
- **服務 down 超過 15 分鐘**：Cowork 可授權緊急部署，事後報告
